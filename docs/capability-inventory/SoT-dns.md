# services/dns capability inventory

Part of the project-wide capability inventory tracked in issue #843. This is the `services/dns` pass — PowerDNS Authoritative + Recursor, `entrypoint.sh`, RPZ generation, and the `services/dns/nats-subscriber` Rust binary (NATS/JetStream consumer, PowerDNS API writer, zone/record rollback listener, TSIG/DDNS config). Also posted as a comment on #843.

Audited against `origin/v0.2.0` (HEAD at audit time: `3f53ac3b`). Cross-referenced against `tests/bats/*.bats`, `scripts/*-simulation.sh`, `docs/dns-admin-ui-scope.md`, `docs/known-good-config-snapshots.md`, and open issues/PRs.

### 1. Container composition (`Dockerfile`, `entrypoint.sh`)

One `dns` image runs three long-lived processes under one `entrypoint.sh`, each in its own restart loop (`run_auth`, `run_recursor`, `run_nats_subscriber`, each `sleep 3` and retry forever): `pdns_server` (authoritative), `pdns_recursor`, and the `nats-subscriber` Rust binary. Multi-stage build: `subscriber-builder` stage (from the shared `build-tools` image, with opt-in sccache/distcc wiring matching the project-wide convention) compiles `nats-subscriber --release`, then the runtime stage installs `pdns-recursor`, `pdns-server`, `pdns-backend-sqlite3`, `dnsutils`, `sqlite3` on `debian:trixie-slim`. Exposes `53/udp`, `53/tcp`, `8081` (auth API), `8082` (recursor API), `8083` (rollback listener, new in #628).

Env vars read by `entrypoint.sh`: `PROXY_IP` (required, no default — fails closed), `PROXY_IPV6`, `PDNS_API_KEY` (default placeholder `CHANGE_ME_PDNS_API_KEY`, rejected at startup if still a placeholder or under 16 chars), `DDNS_ALLOW_FROM` (default `127.0.0.1`), `DDNS_TSIG_KEY`/`DDNS_TSIG_NAME` (default `lancache-ddns-key`)/`DDNS_TSIG_ALGORITHM` (default `hmac-sha256`), `LOG_QUERIES` (falls back to legacy `DNSMASQ_LOG_QUERIES`), `ROOT_ZONE_MIRROR` (default `1`), `NATS_URL`/`NATS_USER`/`NATS_PASSWORD`/`NATS_TOKEN`/`NATS_CONSUMER`/`NATS_RECONCILER`, `KEEP_KNOWN_GOOD_CONFIGS` (default `3`), `DNS_CONFIG_SNAPSHOT_DIR` (default `/var/lib/lancache-dns/config-snapshots`), `DNS_ROLLBACK_LISTEN_ADDR` (default `0.0.0.0:8083`).

**Startup sequence:** render `recursor.conf`/`pdns.conf` from templates (`envsubst`, atomic tmp-file + `mv`) → validate each via the daemon's own `--config=check` → on failure, roll back to the newest known-good snapshot that re-validates, else refuse to start → init SQLite schema if missing → migrate a legacy AAAA-filter marker file to the shared state volume (one-time) → idempotently create the 22 fixed zones (`lan.`, `local.lan.`, 18 per-octet reverse zones, 2 ULA reverse zones) → `configure_ddns_tsig()` (import TSIG key, grant `TSIG-ALLOW-DNSUPDATE` on every DDNS zone) → generate `rpz.zone` from `cdn-domains.txt` with a monotonic SOA serial → start auth server, poll `pdns_control rping` up to 10×0.5s, start recursor, start `nats-subscriber`.

**Fail-closed hardening found in the code, worth noting explicitly:** if `DDNS_TSIG_KEY` is empty or a placeholder, `DDNS_ALLOW_FROM` is forced back to `127.0.0.1` regardless of what the environment/compose requested — closing an unsigned-DNS-UPDATE-from-LAN gap that `dnsupdate-require-tsig=yes` (PowerDNS ≥5.0) would otherwise close directly, but the shipped `pdns-server` (Debian Trixie, 4.9.x) predates that setting.

### 2. PowerDNS Authoritative (`pdns.conf.template`)

`gsqlite3` backend, `local-address=127.0.0.1,${PDNS_LOCAL_ADDRESS}` (dual-bound: loopback for the recursor's own forward, plus this container's real bridge IP — auto-detected every start via `detect_pdns_local_address()`, `ip route get 1.1.1.1 | ... src`, with a same-container fallback scan — so DDNS UPDATE packets from the separate `dhcp` container/host can actually reach it; see #706/#769), `dnsupdate=yes`/`allow-dnsupdate-from=${DDNS_ALLOW_FROM}`, REST API on `:8081` (`webserver-allow-from` covers all three RFC1918 ranges, not just Docker's default pool).

### 3. PowerDNS Recursor (`recursor.conf.template`, `recursor.lua`, `filter-aaaa.lua`)

Forwards `lan.`/`local.lan.`/all 20 reverse zones to the local authoritative server on `127.0.0.1:5300`. Record cache (1M entries, `refresh_on_ttl_perc: 10`, `serve_stale_extensions: 120`), packet cache (500k entries, TTL/negative-TTL/servfail-TTL tuned separately), REST API on `:8082` (used for cache-flush calls). `recursor.lua`: loads the RPZ policy (`rpzFile`), adds negative trust anchors for every locally-forwarded zone (required — DNSSEC validation would otherwise SERVFAIL these, discovered live building #400's test), and conditionally mirrors the root zone via AXFR from three root servers when `ROOT_ZONE_MIRROR!=0` (default on). `filter-aaaa.lua`: a `preresolve` hook that suppresses every AAAA answer globally when `/var/lib/powerdns-state/aaaa-filter-enabled` exists (toggled live by the Admin UI, `dq.variable=true` so it's never cache-poisoned).

### 4. RPZ generation from `cdn-domains.txt` (162 lines / ~140 domains)

Each line becomes `*.<domain> A/AAAA <PROXY_IP>/<PROXY_IPV6>` (plus the bare domain unless prefixed with `.` for wildcard-only), written with a monotonically increasing SOA serial (`date +%s`, bumped by 1 if the clock would otherwise produce a non-increasing value). This is real RPZ policy data (loaded via `recursor.lua`'s `rpzFile`), not a passthrough — the A/AAAA values here directly become the synthesized answer for any matching QNAME.

### 5. Known-good config snapshots — TWO independent mechanisms

**5a. Static-file adapter (shell, `entrypoint.sh`)** — byte-identical copy of `scripts/lib/known-good-snapshots.sh`'s `kgs_*` functions (verified in sync with `services/proxy`/`services/dhcp-proxy`'s copies by `tests/bats/known_good_snapshots_sync.bats`). Snapshots `recursor.conf` and `pdns.conf` independently after each passes its daemon's `--config=check`, retention `KEEP_KNOWN_GOOD_CONFIGS` (default 3, clamped ≥1), atomic staging-dir-then-rename creation. On validation failure, tries stored snapshots newest→oldest until one re-validates; if none do, refuses to start. Includes a stale-`api-key`/stale-`local-address` post-rollback re-stamp (the `PR #769` review follow-up). Test coverage: `tests/bats/dns_known_good_snapshot.bats` (10 tests), `tests/bats/dns_config_snapshot_idempotence.bats` (3 tests), `tests/bats/known_good_snapshots.bats` (11 tests, generic library), `tests/bats/known_good_snapshots_sync.bats` (4 tests). **Fully implemented and tested.**

**5b. Zone/record adapter (Rust, `zone_snapshots.rs` + `rollback_listener.rs`, issue #628)** — a separate mechanism because `nats-subscriber` is a compiled binary in its own process and cannot call the shell library, and zone/record data is API/database-backed (`pdns.sqlite3`), not a file. Manages exactly the 22 `ROLLBACK_ZONES` (`lan.`, `local.lan.`, 18 reverse octet zones, 2 ULA reverse zones — pinned 1:1 against `entrypoint.sh`'s `DDNS_UPDATE_ZONES` by a dedicated unit test, `zone_list_matches_entrypoint_ddns_update_zones`, since there's no cross-language automated check like 5a's bats sync test). Snapshot content = zone's data rrsets with SOA/NS excluded (`filter_data_rrsets` — load-bearing for safety, not just noise reduction: including SOA/NS would let a rollback rewind the SOA serial and would make every live SOA/NS look "extra" and get `DELETE`d). Two independent triggers: (1) post-PATCH hook in `handle_dns_record` (every NATS-applied write), (2) `zone_snapshot_watcher`, a 60s unconditional (not gated by `NATS_RECONCILER`) export-and-diff loop covering Kea's direct-to-PowerDNS DDNS writes, which bypass NATS entirely. Content-diffed via `canonicalize_rrsets` (sorts rrsets by name/type, records by content) so unchanged state never burns through retention. `nats-subscriber` never re-filters — a caller skipping `filter_data_rrsets` would snapshot SOA/NS, which is only enforced by convention/tests, not the type system.

### 6. Zone/record rollback HTTP listener (`rollback_listener.rs`, port 8083)

Runs inside every `dns-*` container (`dns-standard`, `dns-ssl`, and each remote `dns-secondary`), bound `0.0.0.0` (cross-container reachability within the Compose network — not published to the LAN via `ports:`, only `expose:`). Every request requires `X-API-Key: <PDNS_API_KEY>` (constant-time compare against timing side-channels), checked before the request body is even parsed.

- `GET /snapshots` — lists known-good snapshot ids + `created_unix` per zone, newest-first, for all 22 `ROLLBACK_ZONES`.
- `POST /rollback` `{zone, snapshot_id}` — full sequence: auth → zone against the `ROLLBACK_ZONES` whitelist → `snapshot_id` membership check (path-traversal guard) → diff snapshot vs. live (SOA/NS excluded both sides) → `PATCH` the authoritative API → `pdnsutil check-zone` as a post-apply confirmation (informational only — a failure here does **not** revert the already-applied PATCH, by design, since this system has no rollback-the-rollback mechanism) → publish one `lancache.dns.flush` per changed name (reaches both `dns-standard`'s and `dns-ssl`'s recursors via NATS, without this process reaching across containers) → **for `lan.` only**, re-publish the restored rrsets onto `lancache.dns.record` so every NATS-replicated secondary converges immediately instead of waiting for the 60s reconciler → record a fresh known-good snapshot of the now-current (post-rollback) state.

Wired into the Admin UI (`services/ui/src/routes/dns_snapshots.rs`, a thin HTTP forwarder to `DNS_ROLLBACK_URL`, default `http://dns-standard:8083`) — **but scoped to `dns-standard` only today, by explicit documented decision** (`docs/known-good-config-snapshots.md`, `docs/dns-admin-ui-scope.md`). `dns-ssl` and remote `dns-secondary` nodes each run their own identical rollback listener, but the UI has no route pointed at them yet — a `dns-ssl` rollback UI needs its own scoping decision since `local.lan.`/reverse zones aren't NATS-replicated between `dns-standard`/`dns-ssl` in the first place. **No open GitHub issue found tracking this specific "wire the UI to dns-ssl's/secondaries' own rollback listener" gap** — it's documented as a known scope boundary, not tracked as a to-do.

**Test coverage:** `scripts/dns-zone-rollback-simulation.sh` (real end-to-end: two real UI-driven LAN record changes → auto-snapshot verification → direct HTTP calls to `/snapshots` and `/rollback` with real `X-API-Key` auth (including 401-on-missing/wrong-key checks) → real `dig` proof the rollback actually took effect **and** that the post-rollback recursor cache-flush worked (polls against a record with a live 60s TTL already cached with the pre-rollback value)) plus unit tests in `rollback_listener.rs` (4 tests: constant-time compare, API-key check, request deserialization) and `zone_snapshots.rs` (23 tests: zone-list pinning, canonicalization, snapshot CRUD/retention/atomicity, `build_rollback_patch` REPLACE/DELETE semantics). **Fully implemented and well tested**, only the cross-node UI wiring gap above is real, documented debt.

### 7. `nats-subscriber` main loop (`main.rs`)

Connects to NATS (`async_nats::ConnectOptions`, unlimited reconnects, 3s reconnect delay, optional user/password or token auth), creates/gets the `LANCACHE_DNS` JetStream stream (subjects `lancache.dns.>`, file storage, 7-day max age, discard-old), creates/gets a **durable pull consumer** named by the required `NATS_CONSUMER` env var (no default — every node, including each remote secondary, must be given a distinct consumer name so JetStream tracks separate cursors per node).

**Subjects handled by `handle_message`:**
- `lancache.dns.heartbeat*` — ignored, immediately acked.
- `lancache.dns.record` — `handle_dns_record`: parses a `DNSRecord` (`action` replace/delete, `zone`, `name`, `type`, optional `ttl`/`records`), builds a PowerDNS zone-PATCH body, `PATCH`es `http://127.0.0.1:8081/api/v1/servers/localhost/zones/<zone>`. On success, triggers §5b's post-PATCH snapshot. 4xx → ack (unrecoverable, won't retry); 5xx/network error → `RetryStopBatch` (stops consuming the rest of the fetched batch, see below); malformed payload → ack (won't infinite-retry garbage).
- `lancache.dns.flush` — `handle_dns_flush`: parses an optional `{domain}` (falls back to `.` for older publishers), `PUT`s the recursor's `cache/flush?domain=` endpoint. Failure → `RetryContinueBatch` (nak but don't stop the batch — flush has no ordering hazard, unlike record updates).
- Anything else — logged as "Unknown subject", acked.

**Batch/backoff design (issues #87, #653, PR #738):** `consumer.fetch()` uses `no_wait: true` pull mode, so a batch-stop doesn't wait out its own NAK delay before the next fetch. `decide_msg`/`HandleOutcome` is a pure, unit-tested decision function (6 of `main.rs`'s 15 total tests) distinguishing three outcomes: ack, nak-and-continue (flush failures), nak-and-stop-batch (record-update failures — closes a within-batch stale-clobber race where an older update for key X fails transiently while a newer update for the same X later in the same batch would otherwise get applied and acked first). Exponential backoff (1s→30s cap, doubling) on stream errors, fetch errors, or a retryable-batch-stop; resets to 1s on a fully clean batch.

**Known, open, unresolved limitation (issue #772, `known-limitation` label):** a narrower **cross-batch** version of the same race the #738 fix didn't close — an even-newer update for the same key arriving in the *next* fetch (issued immediately, since `fetch()` doesn't wait) can still be acked before the abandoned tail from the previous batch redelivers via `AckWait` and clobbers it. Documented in the code (`main.rs`'s `NakAndStopBatch` comment) and in #772 as accepted, not yet fixed.

**`reconciler()`** — a 60s loop, gated behind `NATS_RECONCILER=1`, that `GET`s the live `lan.` zone and republishes every non-SOA/NS rrset onto `lancache.dns.record` with a stable `reconcile-lan-<name>-<type>` message id (JetStream dedup window absorbs redundant republishes). Every shipped compose file (`deploy/{dev,prod,quickstart,full-setup}/docker-compose.yml`) and `config/{dev,prod}/dns-standard.env` sets `NATS_RECONCILER=1` by default for the `dns-standard` node, so this is effectively always-on in practice, not an opt-in most operators would need to discover. This is the **only** reconciliation path for `lan.`; `local.lan.` and the 20 reverse zones have no NATS reconciliation path at all (documented in `docs/known-good-config-snapshots.md` and `docs/dns-admin-ui-scope.md` — DDNS-populated only, on the primary's own PowerDNS instance, never replicated to secondaries).

**`zone_snapshot_watcher`** — always-on (unlike `reconciler`, not gated by `NATS_RECONCILER`) 60s export-and-diff loop across all 22 `ROLLBACK_ZONES`, feeding §5b's snapshot mechanism; this is the thing that notices Kea's direct-to-PowerDNS DDNS writes since those never touch NATS/this consumer's own subjects at all.

**Test coverage:** unit tests in `main.rs` cover `DNSRecord`/`RRset`/`ZoneInfo` (de)serialization edge cases (missing `changetype` on GET, present on PATCH), `dns_record_to_zone_update` for replace/delete/invalid-action/no-ttl/empty-records, and the full `decide_msg`/batch-stop-vs-continue decision matrix (6 tests including a synthetic multi-message batch simulation). End-to-end: `scripts/ui-nats-dns-integration-simulation.sh` (real UI → real NATS → real `nats-subscriber` → real PowerDNS add/remove, verified via `dig` against both `dns-standard` and `dns-ssl`).

### 8. `nats_publish.rs` — shared publish helper

One function, `publish_dns_record`, factored out so `reconciler()` and `rollback_listener.rs`'s post-rollback `lan.` republish write the exact same `lancache.dns.record` message shape the Admin UI's own publisher and `handle_dns_record`'s consumer side already agree on. No independent tests (covered transitively through the two callers' own tests/simulations); low risk given its size (single JSON-construct-and-publish function).

### 9. TSIG / DDNS

`configure_ddns_tsig()` imports the TSIG key (`pdnsutil import-tsig-key`) and grants `TSIG-ALLOW-DNSUPDATE` on all 22 `DDNS_UPDATE_ZONES` on every start — unconditionally re-run, so it's self-healing for the "key present" case. **Documented, unfixed gap** (`docs/known-good-config-snapshots.md`'s "Zones, records" section): if `DDNS_TSIG_KEY` is later blanked (config edit/migration), `configure_ddns_tsig()` returns immediately without revoking the previously-granted `TSIG-ALLOW-DNSUPDATE` metadata — a host that once had DDNS enabled and then had the key removed can retain stale TSIG authorization indefinitely, and this snapshot/rollback design explicitly does not cover it. No dedicated open issue found for this specific gap (it's flagged in prose only). Reverse-DDNS (`reverse-ddns`) delivery to the per-octet zones was itself broken until #768 (fixed, per-octet `ddns-domains` entries instead of one nonexistent catch-all zone).

**Known, open architectural gap (issue #770):** DHCP-driven DDNS records (Kea forward/reverse DDNS) only reliably reach `dns-standard`, never `dns-ssl`, because Kea's D2 daemon treats its two configured `dns-servers` as failover, not fan-out — `dns-ssl` only receives DDNS updates if `dns-standard` is actually down. Explicitly deferred to v0.3.0 as new architecture work (AXFR replication or routing DHCP events through the existing NATS mechanism), not a narrow bug fix.

PR #730 is open, not yet merged"** and instructs the reader that the linked design doc "still says the topic is deferred." This is now stale: #628 is closed, `zone_snapshots.rs`/`rollback_listener.rs`/`services/ui/src/routes/dns_snapshots.rs` all exist and are implemented/tested (§5b/§6), and `docs/known-good-config-snapshots.md`'s own "Zones, records" section has already been updated to describe it as running behavior ("Issue #628 implemented this design; everything below describes running behavior... not a proposal for a future implementation PR"). `docs/dns-admin-ui-scope.md` was last touched 2026-07-13 (same day #628 closed, but for an unrelated DHCP/PTR fix, #784) and never got its own section-4 update. No open issue found tracking this specific doc-drift; flagging here per this audit's cross-referencing scope rather than fixing it (out of scope for this pure-research pass).
- `docs/architecture-ng.md`'s PowerDNS table lists `FILTER_AAAA_V4`/`FILTER_AAAA_V6` as two separate env vars — neither exists anywhere in `services/dns/` or `services/ui/src`; the real, shipped mechanism is the single global marker-file toggle (§3). Already flagged as stale in `docs/dns-admin-ui-scope.md` itself (not a new finding, restated here for completeness since it's DNS-component-relevant).
- `docs/architecture-ng.md` also claims a "PTR checkbox for LAN IPs" in the Admin UI; no such code exists (`services/ui/src/routes/domains.rs`/`templates/domains.html`) — planned-but-unbuilt, already flagged in the same doc.

### 12. Defined-but-untested / no simulation coverage found

- **Global AAAA-response filter** (`filter-aaaa.lua`, `/domains/aaaa-filter` UI route, `/var/lib/powerdns-state/aaaa-filter-enabled` marker): no `tests/bats/*` file and no `scripts/*-simulation.sh` file references `aaaa-filter-enabled`/`AAAA_FILTER`/`filter-aaaa` at all (only the Lua file itself and the UI route that toggles it). The feature is small and its logic is simple (single marker-file existence check), but it has zero automated end-to-end verification that toggling it actually suppresses AAAA answers via a real `dig` query.
- **Root zone AXFR mirror** (`ROOT_ZONE_MIRROR`, `recursor.lua`'s `zoneToCache` calls): no test file references `ROOT_ZONE_MIRROR` at all — only `config/{dev,prod}/dns-{standard,ssl}.env` (setting the default) and `docs/dns-admin-ui-scope.md`/`entrypoint.sh`/`recursor.lua` themselves. No verification that the root zone actually gets mirrored/cached, or that disabling it (`ROOT_ZONE_MIRROR=0`) actually turns it off.
- **PowerDNS-native secondary/AXFR mode**: confirmed not implemented anywhere (`docs/dns-admin-ui-scope.md` §3a, verified independently during this audit — no `SECONDARY_MASTERS`/`SECONDARY_ZONES` or AXFR-secondary config appears in `pdns.conf.template`/`entrypoint.sh`). Not a gap, a deliberate non-feature: the only secondary mechanism is NATS-based (§7/§8, `lan.` only).
- **RPZ live-reload**: `cdn-domains.txt` add/remove via the UI does not live-reload `rpz.zone` — the recursor only picks up changes on container restart (`recursor.lua`'s `rpzFile()` has no `refresh` polling configured). Documented behavior (`docs/dns-admin-ui-scope.md` §1b), not a bug, but worth noting as a real operational gotcha: the UI's own cache-flush-after-add doesn't make a newly added CDN domain start resolving to the proxy.

### 13. Adjacent open security work (touches `nats-subscriber`'s auth path, not itself part of this component)

Issues #839 (umbrella), #680 (Argon2id password hashing), #681 (active NATS disconnect on secondary removal/rotation), #682 (xkey encryption of secondary credentials in transit) — all about the NATS auth-callout mechanism in `services/ui`/`nats` that issues the credentials `nats-subscriber` connects with, not code inside `services/dns` itself. Listed here only because `nats-secondary-auth-callout-simulation.sh` (§ tests) exercises the real `nats-subscriber` binary as its test client, so a change to any of these would need re-verification against that same script.

### Summary

`services/dns` is one of the more mature components in this codebase: both known-good-snapshot mechanisms (static config files, §5a; zone/record data, §5b) are fully implemented, cross-verified by real end-to-end simulations against live containers (not just unit tests), and the design docs are largely kept in sync with the code (one stale section found, §11). The two genuinely open gaps are already tracked as issues (#772 cross-batch race, #770 dns-ssl DDNS fan-out, #836 setup.sh CLI completion) and explicitly deferred to v0.3.0 rather than hidden. The two smaller, previously-undocumented gaps found in this pass are the AAAA-filter/root-zone-mirror test coverage gap (§12) and the `docs/dns-admin-ui-scope.md` §4 doc-drift (§11) — neither has an open issue yet.
